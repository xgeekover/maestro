결함을 주입한 뒤 시스템의 핵심 보장(**격리·복원·관측 정확성·처리량**)을 실측한다. 모든 시나리오는 실제 독립 JVM 러너 프로세스로 수행된다.

## 시나리오 & 실측 결과 (전부 통과, 총 5)

| # | 결함 주입 | 검증 보장 | 실측 결과 |
|---|---|---|---|
| 1 | **onTick 예외**(매 tick throw) | 격리 + 대시보드(메트릭) 정확성 | 결함 프로세스 errorCount=**7**(봉쇄, RUNNING 유지), 건강 프로세스 **3개 격리**(tick 계속, errorCount=0) |
| 2 | **러너 `kill -9`**(돌연사) | 감시자 재시작 + 격리 | victim **restartCount=1**(자동 복구·RUNNING 복귀), survivor +**10 tick** 진행(무영향) |
| 3 | **무한루프(행)** onTick `while(true)` | tick 워치독 바운드 종료 + 격리 | tick 워치독(800ms)이 감지 → 엔진 **자가 종료(status=ERROR)**, 영구 정지 없음, 건강 프로세스 격리 |
| 4 | **OOM** `new int[~2GB]`/-Xmx32m | 봉쇄(catch) + 격리 | OOM이 tick 에러로 **9회 봉쇄**(프로세스 생존), 건강 프로세스 격리 |
| 5 | **플로우 처리량** producer(20ms)→consumer | 라우팅 처리량·드롭 | **50 msg/s**(2.5초간 125건 전달, **drop=0**) |

> 수치는 실행 시점 측정값(머신·타이밍에 따라 변동). 재현: `./gradlew :backend:test --tests '*SimulationTest'`.

## 관찰 / 결론

- **결함 격리(FR-5)**: process-per-script 경계 덕분에 한 프로세스의 예외·OOM·행·돌연사가 **다른 프로세스에 영향 없음**을 4개 시나리오에서 일관되게 확인.
- **복원(NFR-4)**:
  - **돌연사(kill -9)** → 감시자가 감지하여 **exponential backoff 재시작**(restartCount 증가, RUNNING 복귀).
  - **행(hang)** → 엔진 tick 워치독이 바운드 시간 내 감지하여 **자가 종료(ERROR)**. 좀비 JVM 방지를 위해 러너는 엔진 종료 시 `System.exit`로 확실히 종료(본 단계에서 보강).
- **봉쇄(NFR-3)**: catch 가능한 OOM·tick 예외는 **격리된 tick 에러로 봉쇄**되어 프로세스가 한 결함으로 전체 정지하지 않음(정책 CONTINUE).
- **관측 정확성(FR-6)**: 메트릭(errorCount·tickCount)이 결함/건강 프로세스를 **정확히 구분**해 대시보드 데이터로 반영.
- **처리량(NFR-5)**: 백엔드 릴레이 라우팅이 producer 발신율(50/s)을 **무손실 전달**(현 규모).

## 본 단계 보강 (코드)
- 러너 연결 모드 종료 시 **`System.exit(0)`** — 스크립트 스레드가 행(hang)으로 남아도 프로세스를 확실히 종료(자가 종료 경로 신뢰성↑).

## 한계 / 이월
- **재시작 정책 적용 범위**: 현재 **돌연사**(상태 RUNNING 중 프로세스 사망)만 자동 재시작. **자가 종료(ERROR/STOPPED)** 는 종료로 간주(재시작 안 함). "ERROR 자동 재시작" 옵션은 정책으로 추가 가능(향후).
- **부하/소크(장시간) 테스트**: 본 하니스는 짧은 시나리오. 수백 프로세스·장시간 소크·고처리량 벤치마크는 별도 부하 환경에서 측정(목표 D1: 100–500 동시).
- **고처리량 백프레셔**: drop-oldest는 단위 테스트로 검증(Phase 8). 실부하 드롭률 측정은 부하 환경 과제.
