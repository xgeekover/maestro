## 0. 미션 한 줄

순수 자바 코드를 **동적 스크립트**로 작성·컴파일·실행하고, 각 스크립트를 **격리된 병렬 프로세스**로 돌리며, **노드레드식 플로우**로 연결·분산하고, **대시보드**로 상태·리소스를 관측하는 멀티플랫폼 플랫폼.

## 1. Phase 0 확정 결정 (§6)

§6 미해결 결정 5종에 대한 사용자 확정 답변. 근거·트레이드오프는 [ADR-0001](adr/0001-phase0-foundational-decisions.md)에 기록.

| # | 결정 항목 | 확정값 | 1차 영향 |
|---|---|---|---|
| D1 | 동시 실행 목표 규모 / 호스트 | **수백 개(~100–500)** · 서버급(다중 코어·대용량 RAM) | 프로세스 풀·백프레셔·리소스 제한이 핵심 |
| D2 | 스크립트 신뢰 모델 | **신뢰된 사용자만(내부 도구)** — 경량 샌드박스 | 프로세스 격리 + 리소스 제한 위주, 커널 샌드박스(seccomp) 보류 |
| D3 | 영속 저장소 / 메트릭 보관 | **임베디드 H2 + 인메모리 링버퍼** | 외부 의존 0, 메트릭은 단기 보관 |
| D4 | 배포·인증 형태 | **다중 사용자 + 인증** | 계정/권한·세션 필요 |
| D5 | 플로우 메시지 버스 | **백엔드 릴레이** | 별도 브로커(Redis/Kafka) 미도입 |

> ⚠️ **주의(D3↔D4 긴장)**: "다중 사용자+인증"과 "임베디드 H2"는 일반적으로 함께 쓰기 드문 조합(멀티유저는 보통 Postgres). 본 프로젝트는 **단일 백엔드 인스턴스 + H2 서버 모드**로 시작하되, 동시 쓰기 부하/HA 필요 시 **Postgres 마이그레이션 경로**를 열어둔다. → ADR-0001 §리스크, 미해결 결정 O-1 참조.

| UC-10 | 관리자 | 사용자 계정 로그인, 권한에 따른 스크립트/플로우 접근 제어 | NFR(인증, D4) |

### 핵심 라이프사이클 보장 (FR-3 — Definition of Done의 중심)
- `onStart()`: 프로세스당 **정확히 1회**.
- `onTick()`: 지정 주기마다 반복. tick 단위 예외는 **격리**(정책 `continue`/`stop`).
- `onEnd()`: 정상 종료·사용자 중지·에러 종료 모두에서 **정확히 1회**(best-effort 명시 — `kill -9`/OOM 시 보장 불가, §위협모델 참조).

## 3. 용어집 (Glossary)

| 용어 | 정의 |
|---|---|
| **스크립트(Script)** | `sdk`의 `Script` 베이스를 상속한 순수 자바 클래스. `onStart/onTick/onEnd` 구현. |
| **러너(Runner)** | 스크립트 1개를 담당하는 **독립 JVM 프로세스**. 동적 컴파일 + 라이프사이클 + IPC 수행. |
| **오케스트레이터(Backend)** | Spring Boot. 러너 기동/감시/재시작·스케줄·API·메트릭·플로우 라우팅 담당. |
| **틱(Tick)** | `onTick`의 1회 실행. 주기(period)는 실행 시 지정. |
| **라이프사이클(Lifecycle)** | `onStart → (onTick)* → onEnd` 상태 흐름. |
| **격리(Isolation)** | OS 프로세스 경계. 한 러너의 크래시/OOM/행이 타 러너·백엔드에 무영향. |
| **플로우(Flow)** | 노드(스크립트/모듈)를 엣지(메시지 경로)로 연결한 그래프. node-RED식. |
| **노드(Node)** | 플로우 내 실행 단위(스크립트 또는 모듈 인스턴스). |
| **포트(Port)** | 노드의 입/출력 채널 이름. `emit(port, msg)` / `onMessage(port, h)`. |
| **모듈(Module)** | 입출력 포트·버전·파라미터 스키마를 가진 재사용 패키지. 다중 인스턴스화 가능. |
| **릴레이(Relay)** | 백엔드가 메시지를 직접 라우팅하는 방식(외부 브로커 미사용, D5). |
| **워치독(Watchdog)** | tick 행(hang) 감지·프로세스 사망 감지 메커니즘. |
| **백프레셔(Backpressure)** | 하류가 처리 못 따라갈 때 상류를 늦추는 흐름 제어. |
| **진단(Diagnostic)** | Monaco+JDT LS의 작성 시점 / 컴파일러의 컴파일 시점 오류·경고. |

## 4. 제약 사항 (Constraints)

- **C-1 기술 스택 확정**: Java 21 + Spring Boot(백엔드), process-per-script + `javax.tools.JavaCompiler`(런타임), Electron+React+Monaco+Eclipse JDT LS(데스크탑), React Flow(플로우), Gradle 멀티모듈 + pnpm(빌드), GitHub Actions(CI). 변경 시 ADR 필수.
- **C-2 모노레포 구조 고정**: `sdk/ runner/ backend/ protocol/ desktop/ docs/ deploy/`.
- **C-3 멀티플랫폼**: 데스크탑 Win/Mac/Linux, 백엔드 컨테이너(NFR-1).
- **C-4 격리 단위 = OS 프로세스**: 스레드 기반 격리는 요구사항 미충족(FR-5).
- **C-5 라이프사이클 계약**: `onStart` 1회 / `onTick` 주기 / `onEnd` 1회 — 테스트로 증명(DoD).
- **C-6 D1~D5 결정 준수**: §1 확정값을 설계 기준으로 삼음.
- **C-7 보안은 상시 고려**: NFR-3을 코드 작성 내내 반영(신뢰 모델 D2 기준 경량).

## 5. 동시성 목표치 & 용량 계획 (D1 기반)

- **목표**: 동시 **100–500개** 러너 프로세스를 단일 서버급 호스트에서 안정 운용.
- **프로세스당 비용 가정(설계에서 측정·정밀화)**:
  - JVM 기동 오버헤드 + 기본 heap. → **러너 JVM은 경량 옵션**(`-Xmx` 소값, 필요 시 클래스데이터공유/CDS, AppCDS 검토)으로 메모리 footprint 최소화.
  - 500 × (heap + 메타스페이스 + 스택)이 호스트 RAM에 들어가야 함 → Phase 1에서 **프로세스당 리소스 한도** 산정.
- **설계 함의**:
  - **프로세스 풀/큐잉**: 동시 기동 폭주 방지(스로틀링). 
  - **백프레셔·tick 워치독**: 행 감지 시 격리(NFR-4).
  - **메트릭 수집 부하**: 500 프로세스의 주기 보고가 백엔드/IPC를 포화시키지 않도록 보고 주기·배치 설계.
  - **포트/채널 관리**: 프로세스당 IPC 채널 1개 → fd/포트 한도 확인(ulimit).
- **측정 항목(Phase 9 시뮬레이션 목표)**: 동시 N개 안정 기동 상한, 기동 지연, tick 지터, 플로우 메시지 처리량/지연.

## 6. 위협 모델 초안 (Threat Model — D2: 신뢰된 사용자, 경량 샌드박스)

> 신뢰 모델이 "신뢰된 사용자(내부 도구)"이므로 **악의적 공격자 방어**보다 **사고·실수·자원 폭주로부터의 보호**에 무게. 완전 샌드박스(seccomp/cgroup 강제, 네트워크 차단)는 보류하되 설계에서 확장 훅을 남긴다.

| ID | 위협 | 시나리오 | 완화(Phase) |
|---|---|---|---|
| T-1 | 자원 고갈(메모리) | 스크립트가 무한 할당 → OOM | 프로세스당 `-Xmx` 한도, OOM 시 프로세스만 사망·격리·재시작 정책 (P3/P4) |
| T-2 | 자원 고갈(CPU) | 무한루프 → CPU 점유 | tick 워치독(행 감지)·프로세스 CPU 한도(cgroup/nice 검토)·강제 종료 (P4/P9) |
| T-3 | 행(hang)·교착 | onTick이 영원히 미반환 | tick 타임아웃 워치독 → ERROR 전이 → onEnd → 재시작 (P3/P4) |
| T-4 | 프로세스 크래시 | 러너 비정상 종료(`kill -9`) | 백엔드 감시 → 재시작 정책. `kill -9`/OOM에서 `onEnd`는 **보장 불가**(best-effort 명시) (P4) |
| T-5 | 파일/네트워크 접근 | 스크립트가 임의 파일·소켓 접근 | 신뢰 모델상 허용(기본). **허용목록 정책 훅**을 남겨 차후 강화 가능 (설계 P1) |
| T-6 | 비밀정보 노출 | 환경변수/시크릿이 러너에 주입 | 러너에 **비밀정보 비주입**, 최소 권한 환경으로 기동 (P1/P4) |
| T-7 | 컴파일 시 자원 남용 | 거대/악성 소스로 컴파일러 자원 폭주 | 소스 크기 한도·컴파일 타임아웃·컴파일 산출물 캐시 (P3) |
| T-8 | 인증/인가 우회 | 다중 사용자 환경에서 타 사용자 리소스 접근(D4) | 인증 + 리소스 소유권 기반 인가 (P1 설계, P4/P5 구현) |
| T-9 | IPC 신뢰 경계 | 러너↔백엔드 채널 위변조 | 로컬 전용 바인딩·프로세스 토큰/핸드셰이크 (P1/P4) |
| T-10 | 클래스로더 누수 | 동적 클래스 로드 반복 → 메타스페이스 누수 | process-per-script로 프로세스 종료 시 회수(설계상 자연 완화) (P3) |

**보안 원칙(NFR-3)**: 프로세스당 리소스 제한, 최소 권한, 비밀정보 비주입, (확장 가능한) 네트워크/파일 정책 훅. 신뢰 모델 변경 시 ADR로 격상.

## 8. 리스크 & 가정

- **R-1**: 수백 JVM의 메모리 footprint가 호스트 한계를 칠 수 있음 → O-3(AppCDS)·Phase 9 측정으로 검증. *가정: 서버급 호스트(예: 32GB+ RAM, 8+ 코어).*
- **R-2**: `onEnd` 1회 보장은 graceful 경로에서만 — `kill -9`/OOM/전원차단은 best-effort. DoD 문구에 명시 필요.
- **R-3**: JDT LS를 SDK 클래스패스와 연동하는 작업(Phase 5)이 난이도 높음 → 조기 스파이크 권장.
- **R-4**: H2 다중 사용자 동시성(O-1) → 데이터 접근 추상화로 전환 리스크 흡수.
- **R-5**: 동적 컴파일 + 커스텀 ClassLoader의 메타스페이스/캐시 관리 → process-per-script가 자연 완화하나 컴파일 캐시 무효화 정확성 필요(소스 해시 기반).

**전역 가정**: 단일 백엔드 인스턴스로 시작(HA 비요구), 호스트는 서버급, 사용자는 신뢰됨, 네트워크는 로컬/사내.
