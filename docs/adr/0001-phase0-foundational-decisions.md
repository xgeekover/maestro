# ADR-0001 — Phase 0 기반 결정 (Foundational Decisions)

- 상태: **승인 대기(Proposed)** — Phase 0 게이트에서 확정
- 날짜: 2026-06-28
- 맥락: [docs/00-analysis.md](../00-analysis.md) §1, §6

## 맥락 (Context)

빌드 프롬프트 §6은 전체 아키텍처를 좌우하는 5개 결정을 Phase 0에서 사용자와 확정하도록 요구한다. 이 ADR은 그 결정과 근거·트레이드오프·결과를 기록한다.

## 결정 (Decisions)

### D1. 동시 실행 목표 규모 — **수백 개(~100–500), 서버급 호스트**
- **근거**: 단일 개발기(수십)보다 큰 운영 규모를 목표하되, 분산 클러스터(수천+)의 복잡도는 회피.
- **결과**: 프로세스 풀·기동 스로틀링·백프레셔·프로세스당 리소스 한도·메트릭 보고 배치가 설계 1순위. 단일 호스트 가정.

### D2. 신뢰 모델 — **신뢰된 사용자만(내부 도구), 경량 샌드박스**
- **근거**: 내부/사내 도구. 악의적 임의코드 방어보다 사고·자원 폭주 보호에 집중.
- **결과**: 프로세스 격리 + 리소스 제한(`-Xmx`, ulimit/cgroup 검토) 위주. seccomp/네트워크 차단 등 커널 샌드박스는 **보류하되 정책 훅을 설계에 남김**. 신뢰 모델 강화 시 새 ADR로 격상.

### D3. 영속 저장소 / 메트릭 — **임베디드 H2 + 인메모리 링버퍼**
- **근거**: 외부 의존 0, 로컬·간편 운영. 메트릭은 단기 관측 용도.
- **결과**: 데이터 접근은 Spring Data로 추상화. 메트릭은 프로세스당 최근 N분 링버퍼 → 대시보드 WS 푸시.

### D4. 배포·인증 — **다중 사용자 + 인증**
- **근거**: 여러 사용자가 공유하는 백엔드. 계정/권한/세션 필요.
- **결과**: 로컬 계정(BCrypt) + 세션/JWT로 시작(O-8), 리소스 소유권 기반 인가. SSO/OIDC는 후순위.

### D5. 플로우 메시지 버스 — **백엔드 릴레이**
- **근거**: 수백 규모·초기 단계에서 외부 브로커 불필요.
- **결과**: 백엔드가 노드 emit → 하류 노드로 직접 라우팅. 처리량 한계 도달 시 브로커(Redis/Kafka) 도입을 새 ADR로 검토.

## 결과 & 리스크 (Consequences & Risks)

- ⚠️ **D3↔D4 긴장**: 다중 사용자+인증은 통상 Postgres와 함께 쓰나 H2를 선택했다. **단일 백엔드 인스턴스 + H2 서버 모드**로 시작하고, 데이터 접근 추상화(Spring Data)로 **Postgres 전환 경로**를 확보한다. 동시 쓰기 병목/HA 요구 시 ADR로 전환. → 미해결 결정 **O-1**.
- 수백 JVM의 메모리 footprint 리스크(R-1) → AppCDS·경량 heap(O-3) + Phase 9 측정.
- `kill -9`/OOM에서 `onEnd` 보장 불가(R-2) → DoD에 best-effort 명시.

## 대안 (Alternatives Considered)
- 동시성: 수십(과소)·수천+분산(과복잡) → 기각, 수백 채택.
- 신뢰모델: 완전 샌드박스 → 현 단계 과투자로 보류(훅만 유지).
- 저장소: Postgres/시계열 DB → 초기 운영 단순성 위해 보류(전환 경로 유지).
- 버스: Redis/Kafka → 초기 불필요, 후속 검토.
